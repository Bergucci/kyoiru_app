## 19. Step 2 制約分担

### 19.1 DB 制約で担保する項目

- `users.user_id` の unique
- `auth_identities` の `(provider, provider_subject)` unique
- `friendships` のユーザーペア一意
- `user_blocks` の `(blocker_user_id, blocked_user_id)` 一意
- `friendship_requests` の主キー整備と参照整合
- 外部キー整合
  - request sender / receiver
  - friendship user pair
  - block blocker / blocked
- 必須カラムの null 禁止
- enum カラムの型固定

理由:
- 破れるとデータ破損になる
- アプリケーションバグで重複や孤児レコードが入るのを防ぐ

### 19.2 DB 制約だけでは担保しない項目

- 1 日 30 件の友達申請上限
- 拒否後 30 日再申請不可
- 拒否後 24 時間以内の拒否取り消し
- block 実行時の副作用
  - 既存友達解除
  - pending 無効化
  - 見守り停止
- ID 検索公開設定による検索結果除外
- block 優先の横断遮断
- JST 朝 6:00 business day 判定

理由:
- 時刻依存、状態遷移依存、複合判定が多い
- DB 制約に押し込むと可読性と保守性が落ちる

### 19.3 DB とサービス層の併用で担保する項目

- pending 重複申請禁止
  - DB: 同一 sender / receiver の active pending を一意にする
  - サービス層: block、再申請制限、自己申請禁止を事前判定
- ユーザーID変更制限
  - DB: `user_id_changed_at` を保持
  - サービス層: 30 日判定と次回変更可能日算出
- 初回プロフィール完了条件
  - DB: `profile_status`
  - サービス層: ホーム遷移ガード

### 19.4 Step 2 レビュー時に確認する項目

- DB に入れるべき制約がサービス層に逃げていないか
- サービス層で扱うべき時刻依存ロジックを DB に押し込んでいないか
- unique 制約の対象が仕様と一致しているか
- 将来の Step 3、Step 4 で必要なカラムが不足していないか

## 23. Step 4 block 副作用の適用タイミング

### 23.1 基本方針

- block は `実行成功レスポンスを返す前` に必要副作用を完了させる
- 「block 自体は成功したが、友達解除や見守り停止は後で行う」という非同期前提にしない
- block 成功時点で、接触導線は即時閉じている状態を正とする

### 23.2 同期で即時適用する副作用

- block レコード作成
- 既存友達関係の解除
- pending 友達申請の無効化
- 見守り設定の停止
- ID 検索結果からの除外
- 招待候補 / おすすめ導線からの除外

### 23.3 画面反映のタイミング

- block API 成功後、クライアントは関連キャッシュを即 invalidate する
- 最低限再取得対象に含めるもの
  - block 一覧
  - 友達一覧
  - 友達申請一覧
  - グループ一覧
  - グループ詳細
  - 見守り対象一覧

### 23.4 非同期でよいもの

- 監査ログの二次集計
- 不正利用検知の集計
- 通知配信履歴の後処理

注意:
- 非同期でよいのは補助ログだけ
- 接触遮断に関わる本体処理は同期で終える

### 23.5 失敗時の扱い

- 友達解除、pending 無効化、見守り停止のいずれかに失敗した場合は block 成功扱いにしない
- 可能であれば 1 トランザクションで処理する
- 部分成功をユーザーに見せない

### 23.6 Step 4 完了判定への追加条件

- block 実行直後に友達一覧から相手が消える
- block 実行直後に pending 申請が無効化される
- block 実行直後に見守り対象から外れる、または停止状態になる
- 既存グループでは同一画面再取得時に最小表示へ切り替わる

- 部分成功を許す設計になっていないか
- 既存グループだけは削除ではなく最小表示に落ちる設計になっているか

## 25. Step 5 グループ・招待・生存報告で固定する項目

### 25.1 Step 5 の対象

- グループ作成
- グループ招待リンク発行 / 再発行 / 参加
- グループ一覧 / グループ詳細の状態表示
- 今日の生存報告
- 気分スタンプ

### 25.2 Step 5 完了判定

- グループ作成時に `グループ名`、`利用タイプ`、`グループアイコン` を設定できる
- 初期メンバー選択対象は友達のみで、block 関係の相手は候補に出ない
- グループ招待リンクは `30 日`、`1 回のみ`、`再発行で旧リンク即無効` が成立する
- 期限切れ、使用済み、block 関係では参加不可になる
- JST 朝 6:00 基準で business day が切り替わる
- 生存報告は 1 日 1 回のみで、2 回目以降は送れない
- 同日 2 回目以降でも、未設定なら気分スタンプだけは設定できる
- 気分スタンプは生存報告後のみ設定可能で、削除後の再設定は不可
- 過去 7 日分の履歴が確認でき、過去日は編集不可
- グループ一覧で `反応済み人数 / 未反応人数 / 最も強い状態ラベル` が出る
- グループ詳細でメンバー順が `自分 → monitor_alert / overdue → pending → checked_in` になる
- block 済みメンバーは既存グループ内で最小表示に落ち、タップ不可になる

### 25.3 Step 5 で実装担当に提出させるもの

- グループ作成、グループ参加、招待リンク再発行、生存報告、気分スタンプの動作確認結果
- 招待リンクの `期限切れ / 使用済み / 再発行後旧リンク / block 関係` の拒否確認結果
- JST 朝 6:00 をまたぐ生存報告判定の確認結果
- 同日 2 回目送信不可と、スタンプ未設定時のみ追加可能である確認結果
- スタンプ削除後の再設定不可の確認結果
- block 済みメンバーが既存グループで最小表示になる確認結果

### 25.4 Step 5 で差し戻す条件

- 端末ローカル日付で判定している
- 招待リンクが複数有効になり得る
- 生存報告の 1 回目時刻を 2 回目で上書きする
- スタンプ削除後に同日再設定できてしまう
- block 済み相手が新規グループ作成や新規招待で混入できる
- 既存グループ内で block 済み相手に詳細遷移できる

## 26. Step 5 承認時に見るポイント

- 招待リンクの有効条件と無効条件が仕様どおりに閉じているか
- business day 判定が API 側で一元化されているか
- 生存報告と気分スタンプの関係が分離されつつ矛盾していないか
- グループ一覧の集計と、グループ詳細の並び順が別要件として正しく扱われているか
- block 済み相手への接触導線が既存グループでも閉じているか

## 27. Step 6 通知・課金で固定する項目

### 27.1 Step 6 の対象

- グループごとの通知温度感設定
- 無料文脈の未反応通知
- RevenueCat entitlement
- Billing Grace Period
- 有料機能の停止 / 復元条件

### 27.2 Step 6 完了判定

- 通知温度感はグループごとに `ゆるい / ふつう / 気にかける` を設定できる
- `ゆるい` はプッシュ通知なし、アプリ起動時確認のみになる
- `ふつう` は翌日朝 6:00 を過ぎても未反応なら 1 回だけ軽い通知を送る
- `気にかける` は当日 21:00 に軽い通知、翌日朝 6:00 に再通知を送る
- 22:00〜7:00 は原則通知しない
- 見守りモードの段階通知は、無料の通知温度感とは別系統で扱う
- RevenueCat entitlement により見守りプランの有効 / 無効を判定できる
- 解約後も store 有効期限満了までは有料機能を使える
- Billing Grace Period 中は見守り機能を維持する
- Grace Period 終了後は有料機能を停止する
- 有料機能停止後も見守り設定は保持し、再契約時に復元できる
- 有料機能停止後も位置履歴は復元しない
- 課金訴求は `見守り有効化時 / 未反応継続時 / 親・高齢者用途文脈 / 解約後` の必要時だけに出る

### 27.3 Step 6 で実装担当に提出させるもの

- 通知温度感ごとの通知発火確認結果
- 22:00〜7:00 の通知抑止確認結果
- 無料通知と見守り段階通知が別条件で発火する確認結果
- entitlement 有効、Grace Period 中、Grace Period 終了後、解約済み期限内の各状態での機能可否一覧
- 再契約時に見守り設定が復元し、位置履歴は復元しない確認結果
- 課金訴求が常設バナー化していない確認結果

### 27.4 Step 6 で差し戻す条件

- 無料通知と見守り段階通知が同じロジックや同じ設定で処理されている
- 22:00〜7:00 に無料通知が飛ぶ
- entitlement 失効直後に Grace Period を無視して停止する
- Grace Period 終了後も有料機能が残る
- 課金停止時に見守り設定まで削除してしまう
- 再契約時に古い位置履歴まで復元してしまう
- 必要時以外でも強い課金訴求が常設表示される

## 28. Step 6 承認時に見るポイント

- 通知温度感がグループ単位になっているか
- 無料の未反応通知と、有料の段階通知が責務分離されているか
- RevenueCat entitlement と store 有効期限、Grace Period の扱いが矛盾していないか
- 停止するのが `有料機能` であって、`見守り設定データ` ではないことが守られているか
- 課金導線が必要時表示に留まり、押し売りになっていないか

## 29. Step 7 見守りで固定する項目

### 29.1 Step 7 の対象

- 見守り有効化リクエスト
- 見守られる側の同意 / 同意撤回
- GPS 共有設定
- 緊急連絡先
- 段階通知
- 複数回チェックイン設定
- 見守りダッシュボード

### 29.2 Step 7 完了判定

- 見守りは相手単位で設定できる
- 見守る側の課金が有効でも、見守られる側の明示同意完了までは有効化扱いにしない
- 見守られる側が無料会員でも見守り対象になれる
- GPS 共有設定は `共有しない / 未反応時のみ参照可能 / 常時共有` の 3 つで扱う
- 初期値は `未反応時のみ参照可能` になる
- 位置参照は GPS 設定と段階条件に従って制限される
- 緊急連絡先は `1件のみ / 氏名必須 / 電話番号必須 / 続柄任意` で扱う
- 緊急連絡先は第 3 段階到達時のみ表示される
- 緊急連絡先表示時は閲覧ログを残す
- 段階通知は `当日 21:00 / 翌日 6:00 / 翌日 12:00` の 3 段階で動く
- 第 2 段階で位置確認導線が出る
- 第 3 段階で緊急連絡先と「連絡を取ってみてください」導線が出る
- 複数回チェックインは `1回 / 2回 / 3回` から選べる
- 時間帯テンプレは `朝 / 朝・夜 / 朝・昼・夜` で扱う
- 見守りダッシュボードで `対象一覧 / 今日の反応状況 / 最終位置 / 最終反応時刻 / 緊急連絡先設定有無 / 複数回チェックイン達成状況 / エスカレーション発生状況` が見える
- block 発生時は見守り設定より block を優先し、見守りは停止状態になる
- block 解除後も自動再開せず、再設定が必要になる
- 同意撤回時は見守りを即停止し、位置収集も停止する

### 29.3 Step 7 で実装担当に提出させるもの

- 見守り開始リクエスト送信から同意完了までの動作確認結果
- 同意未完了では有効化扱いにならない確認結果
- 同意撤回時の停止確認結果
- GPS 共有 3 設定ごとの位置参照可否確認結果
- 段階通知の `21:00 / 翌日 6:00 / 翌日 12:00` 発火確認結果
- 第 2 段階で位置確認導線のみ、第 3 段階で緊急連絡先表示まで進む確認結果
- 緊急連絡先の設定 / 更新 / 削除 / 表示ログ確認結果
- 複数回チェックイン設定ごとの達成状況反映確認結果
- block 時に見守り停止し、block 解除後も自動再開しない確認結果

### 29.4 Step 7 で差し戻す条件

- 課金有効だけで同意前に見守りが ON になる
- 無料ユーザーを見守り対象にできない
- GPS 設定が 3 段階でなく単純 ON / OFF に簡略化されている
- 第 2 段階前に位置確認できる
- 第 3 段階前に緊急連絡先が見える
- 緊急連絡先を複数件保持している
- block 時に見守りが残る、または block 解除で自動再開する
- 同意撤回後も位置収集や位置参照が残る

## 30. Step 7 承認時に見るポイント

- 課金権利と見守り同意が別条件として扱われているか
- 位置参照条件が `同意 / GPS設定 / 段階条件 / block` の掛け合わせで閉じているか
- 緊急連絡先の表示条件が最終段階限定で守られているか
- 複数回チェックインが見守り文脈の拡張機能として分離されているか
- block 優先と同意撤回時停止が、見守りフロー全体で崩れていないか
